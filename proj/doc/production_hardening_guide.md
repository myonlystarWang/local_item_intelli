# 精密工具智能化管理系统生产落地加固指南

* **版本**：V2.0
* **日期**：2026-06-22
* **审查范围**：`proj/PRD.md`、`doc/系统测试问题优化方案_20260621_V1.0.md`、`proj/server`、`proj/web`、`proj/app`
* **结论**：当前系统已具备“Web 建档/台账、App 离线扫码流转、近场同步、冲突报告”的原型闭环，但距离生产落地仍缺少账号鉴权、设备授权、责任人认证、并发一致性、迁移与运维等关键能力。生产上线前应按本文 P0/P1 项逐项完成。

## 2. 上线阻断项（P0）

### 2.1 后端鉴权与 Web 登录

**当前问题**

`proj/server/main.py` 所有接口均无鉴权，局域网内任意设备可以创建工具、调整库存、删除字典、伪造同步包。

**生产要求**

1. Web 端增加库管员登录页。
2. 后端增加账号表、密码哈希、登录接口、JWT 签发和校验。
3. 所有写接口必须校验 JWT。
4. App 同步接口至少要校验设备密钥或设备授权令牌。

**建议最小实现**

* `POST /auth/login`：库管员用户名密码登录，返回 JWT。
* Web `Login.vue`：登录后保存 token，Axios 自动附带 `Authorization: Bearer <token>`。
* 后端写接口加 `Depends(require_admin)`。
* `.env` 配置 `JWT_SECRET_KEY`，禁止写死密钥。

### 2.2 操作人身份认证

**当前问题**

App 端身份确认只是从人员/大队字典中选择，任何人都能切换成任何操作人，不足以支撑责任界定。

**生产要求**

1. 每个操作人应有 PIN 或工号绑定。
2. App 首次选择身份后可要求输入 PIN。
3. 设置页切换身份时必须再次验证 PIN。
4. 同步日志中保留 `operator_id`、`operator_name`、`team`，不能只存姓名。

**建议最小实现**

* 后端字典人员扩展为人员表：`id/name/team/pin_hash/is_active`。
* App 下发人员数据时只下发必要字段和 PIN 校验摘要，或由库房端生成本地授权包。
* 本地日志新增 `operator_id` 字段。

### 2.3 设备授权管理

**当前问题**

App 可填写 `terminal_uuid`，但后端 `sync.py` 不校验，Web 也没有启用/禁用设备的页面。

**生产要求**

1. 后端新增 `authorized_devices` 表。
2. `/sync` 处理前校验 `terminal_uuid` 是否存在且启用。
3. Web 新增「终端设备授权」页面，支持登记、启用、禁用、备注设备。
4. App 首次启动自动生成不可重复 UUID，不应默认 `terminal-handheld-001`。

**建议表结构**

```sql
authorized_devices(
  uuid varchar primary key,
  name varchar not null,
  is_active boolean not null default true,
  registered_at timestamp not null,
  last_sync_at timestamp,
  remark text
)
```

### 2.4 同步并发一致性

**当前问题**

`process_offline_sync` 逐条回放日志，最后统一 `commit`。多台设备同时同步时，可能发生库存扣减丢失、使用次数覆盖、状态判断竞态。

**生产要求**

1. 对本次同步涉及的工具和配件加事务锁。
2. 配件库存扣减必须在数据库事务内原子完成。
3. 同步响应必须明确 `success / partial_success / error`。
4. App 不能在有失败日志时无条件清空全部本地日志。

**建议实现**

* PostgreSQL 使用 `SELECT ... FOR UPDATE` 锁定工具和配件行。
* `SyncLogResult` 增加 `local_log_id/tool_code/result_code`。
* App 只删除服务端确认 `success` 的本地日志，`conflict/error` 保留到“人工处理”列表。

### 2.5 数据库迁移管理

**当前问题**

后端依赖 `Base.metadata.create_all()`，只能建新表，无法安全升级生产库。App 端虽然已有 `onUpgrade`，但没有迁移测试和现场数据备份策略。

**生产要求**

1. 后端引入 Alembic 管理所有 schema 变更。
2. App 每次修改 SQLite schema 必须提升版本，并提供迁移测试。
3. 生产升级前必须备份 PostgreSQL 和关键手持端本地库。

**最低要求**

* 后端加入 `alembic/` 和 `alembic_version`。
* 每次变更 `models.py` 必须提交迁移脚本。
* Flutter 增加 `local_db_migration_test.dart`，验证旧库升级不丢数据。

### 2.6 API 版本管理与离线兼容

**当前问题**

App 和 Server 当前使用裸接口路径，如 `/sync`、`/tools`。野外设备可能数周不更新，一旦服务端字段变更，旧 App 可能同步失败。

**生产要求**

1. API 路径加版本：`/api/v1/sync`、`/api/v1/tools`。
2. 同步请求携带 `app_version/schema_version`。
3. 服务端保留至少一个旧版本兼容窗口。
4. 对不兼容版本返回明确错误和升级提示。

### 2.7 环境配置与密钥管理

**当前问题**

`DATABASE_URL` 有默认弱密码，CORS 为 `*`，Web API 地址硬编码 `127.0.0.1:8000`。此外，当前后端在 PostgreSQL 连接失败时会自动降级为本地 SQLite，这对开发友好，但在生产会造成“服务看似正常、数据实际写到本地临时库”的严重风险。

**生产要求**

1. `.env` 注入数据库地址、JWT 密钥、CORS 白名单。
2. Web 使用 `VITE_API_BASE_URL`。
3. CORS 收缩到库房端真实域名/IP，不能 `allow_origins=["*"]`。
4. 生产环境禁止自动降级到 SQLite；连不上 PostgreSQL 应启动失败并报警。
5. `.env`、`*.db`、日志文件必须加入 `.gitignore`。
6. 正式部署建议加 HTTPS/TLS；至少内网 Nginx 反代，避免同步包和人员信息明文传输。

### 2.8 数据模型和同步语义约束

**当前问题**

多个关键字段缺少生产级约束：工具状态是普通字符串，时间混用 `datetime.now()` 与 `datetime.utcnow()`，`parse_time` 失败会静默回退当前时间，接口对寿命阈值、库存数量等字段缺少下限校验，同一离线日志重复上传时也没有幂等保护。

**生产要求**

1. 工具状态改为枚举或数据库 CHECK 约束，仅允许“在库 / 离库 / 地点变更 / 报废”。
2. 全系统统一 UTC timezone-aware 时间。
3. `parse_time` 失败时该条日志应返回 `error`，不能静默用当前时间替代。
4. 字典表增加 `(dict_type, dict_value)` 唯一约束。
5. 输入校验增加下限：`lifespan_limit > 0`、`safety_stock >= 0`、`current_stock >= 0`、`qty > 0`。
6. App 本地日志增加稳定 `local_log_id`，服务端记录已处理日志 ID，重复同步不得重复扣减库存或重复累加寿命。

### 2.9 同步结果语义和失败日志保留

**当前问题**

`/sync` 当前容易把整体状态表达成成功；App 端同步成功后会覆盖本地账本并清空本地日志。生产中如果报告里存在 `conflict/error`，不应把失败记录直接抹掉。

**生产要求**

1. `SyncResponse.status` 区分 `success / partial_success / error`。
2. `SyncLogResult` 增加 `local_log_id`、`tool_code`、`result_code`。
3. App 只删除服务端确认成功的日志。
4. 冲突/失败日志保留到“待人工处理”列表，可导出给库管员核对。

## 3. 正式运行前应完成（P1）

### 3.1 设备丢失与孤儿状态处理

离线架构下，手持机丢失或长时间不回库时，工具可能永久停留在“离库/地点变更”。生产需要 Web 端人工处理流程：

1. 管理员选择工具，查看最后同步设备和操作人。
2. 填写复位原因和审批人。
3. 执行“强制归库/强制锁定/报废”。
4. 写入不可删除审计日志。

### 3.2 报废流程

PRD 当前暂缓报废，但生产资产闭环建议补齐：

* 仅“在库”工具允许报废。
* 已报废工具禁止出库、地点变更、维保。
* 报废需要原因、审批人、附件或备注。
* 报废记录进入工具履历和审计报表。

### 3.3 真实报表导出

当前 Web 只是 `alert` 模拟导出。生产建议至少实现：

1. 工具资产报表。
2. 配件库存报表。
3. 操作日志/审计报表。
4. 支持按日期、人员、大队、状态筛选。
5. 大数据量导出应后端流式生成，不在浏览器一次性拼大数组。

### 3.4 真实 Excel 批量导入

当前 Web 的批量导入是模拟：点击选择文件不会真正解析 Excel，进度条也是前端计时，最终只插入一条随机编码的示例工具。生产需要：

1. 前端使用真实文件选择，限制 `.xlsx` 模板。
2. 后端新增 `POST /tools/batch-import`，解析 Excel。
3. 校验编码唯一、必填字段、寿命阈值、型号等。
4. 返回成功/失败明细，前端展示每条失败原因。
5. 导入操作写入审计日志。

### 3.5 健康检查与运维状态

新增：

* `GET /health`：服务和数据库可用性。
* `GET /version`：后端版本、数据库迁移版本。
* Web 顶部显示 API 连接状态。
* App 同步页显示服务器版本和最近一次成功同步时间。

### 3.6 同步日志生产化

当前 `SyncLog` 已有原型，但生产还需要：

* 分页查询。
* 按设备、人员、结果类型、时间筛选。
* 保留策略，例如保留 180 天。
* 失败日志支持人工标记“已处理”。

### 3.7 移除 Web 写操作 Mock 降级

`web/src/api.ts` 仍保留大量 catch fallback 到内存 Mock。生产中写操作不允许静默降级，否则库管员会以为保存成功但数据没有入库。

要求：

* 读操作失败可展示空状态或离线提示。
* 写操作失败必须弹出错误并中止。
* Axios 增加统一错误拦截器。

### 3.8 归还周期、分队持有量与预警处理

当前 Dashboard 超期判断固定按 30 天计算，但 App 出库表单已经支持 7-90 天归还期限。生产应把实际 `return_days` 写入服务端工具状态，例如 `expected_return_days`，Web 按该字段计算超期。

同时 PRD 要求的“分队持有量视图”和“预警快捷处理窗口”尚未实现。建议补充：

* `tools.team` 当前持有大队字段。
* Dashboard 按大队聚合工具持有量。
* 预警卡片支持查看详情、发起核验、标记处理。

## 4. 稳定运行增强项（P2）

### 4.1 App 分发与更新

建议建立内部发布机制：

* Debug 包仅用于开发。
* 生产使用签名 APK。
* 维护版本号、发布日期、兼容 API 版本。
* 手持机回库时检查是否有新版本。

### 4.2 App 本地数据保护

生产手持机可能丢失，应考虑：

* SQLite 加密。
* 设置页敏感配置隐藏或加 PIN。
* 设备停用后服务端拒绝同步。
* 长时间未同步设备进入风险清单。

### 4.3 性能与分页

当前 `/tools` 返回全量工具并带履历，规模上来后会变慢。建议：

* `/tools` 只返回列表字段并分页。
* `/tools/{code}/histories` 独立返回履历。
* `/sync` 改为增量返回，或支持按 `last_sync_time` 拉取变更。
* 对 `tools.status`、`tool_histories(tool_code,timestamp)`、`sync_logs(timestamp)` 建索引。

### 4.4 审计日志

除同步日志外，以下 Web 写操作也要进审计：

* 新建/编辑工具。
* 新建/调整配件库存。
* 新增/修改/删除字典。
* 设备启用/禁用。
* 强制复位和报废。

审计字段至少包含：操作人、时间、IP、动作、对象、旧值、新值、结果。

### 4.5 备份与恢复

生产必须明确：

* PostgreSQL 每日备份。
* 备份保留周期。
* 恢复演练步骤。
* 手持端未同步数据丢失时的人工补录流程。

### 4.6 App 发布平台差异与本地日志兜底

如果未来需要 iOS 或多平台分发，需要补齐平台权限说明，例如 iOS 的 `NSCameraUsageDescription`。Android 生产包则要建立签名、版本号、安装手册和回滚包。

移动端还应提供本地待同步日志导出能力。设备损坏但仍可读取文件时，至少能导出 JSON/CSV 给库管员人工导入或核对。

### 4.7 测试体系清理

当前 `proj/app/test/widget_test.dart` 仍是 Flutter 计数器脚手架测试，与当前业务页面不匹配。生产前应删除该测试，改为：

* App 离线状态机测试。
* SQLite 迁移测试。
* 未登记资产拦截测试。
* 寿命超限禁止出库测试。
* 后端重复出库、配件不足、非法状态流转测试。

### 4.8 业务统计字段补齐

为减少运行期报表聚合成本，建议补充：

* `tools.team`：当前持有大队。
* `accessories.total_consumed`：累计消耗量。
* `tools.expected_return_days`：本次出库预计归还天数。
* `audit_logs`：所有 Web 写操作和管理员强制操作。

## 6. 生产上线最小验收清单

上线前至少满足以下检查：

1. Web 端未登录不能访问台账、字典、设备、报表页面。
2. 未授权 `terminal_uuid` 调用 `/sync` 返回拒绝。
3. 两台设备重复离线出库同一工具，只允许第一条成功，第二条冲突留痕。
4. 配件库存不足时，维保同步失败且不扣减库存。
5. App 同步失败时，本地日志不丢失。
6. App 升级 SQLite schema 后，旧数据仍保留。
7. 后端升级后，旧 App 至少能得到明确兼容响应。
8. Web 导出三类报表数据正确，超过 1 万条履历不导致浏览器崩溃。
9. Excel 批量导入能解析真实模板，并返回逐条成功/失败结果。
10. 归还超期预警按实际 `return_days` 计算，不再固定 30 天。
11. 数据库可完成备份和恢复演练。
12. 所有写操作均可在审计日志中追踪到操作人和时间。
